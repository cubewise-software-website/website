## What Arc+ actually is

Important distinction for all copy:

> **"Arc+ is Arc connected to Pulse."**

It is not a separate product download. It is the combined capability you get when both Arc and Pulse are licensed and connected. The `/platform/` page states it explicitly: *"Connect Arc to Pulse to unlock many new features. We call this Pulse-powered mode 'Arc+'."*

**Implication for pricing / sales.** If a buyer asks "how do I buy Arc+?" the answer is: buy Arc and Pulse, and turn on the integration.

## The DevOps lifecycle — how to structure Arc+ copy

The Arc+ hero promises six stages: **Plan. Build. Test. Migrate. Operate. Support.** Every section of an Arc+ page should ladder to one of them.

| Stage | Features that deliver it |
|-------|--------------------------|
| **Plan** | Impact analysis · To Do list · Dependency visualisation |
| **Build** | Best-practice rules · Code templates · Centralised platform |
| **Test** | Unit testing · Code coverage · Test collections |
| **Migrate** | Deployment packages · Live migration · Migration logs · Rollback |
| **Operate** | Licence compliance · Cross-instance dashboard · Historical lookback |
| **Support** | Live session monitoring · CPU/memory/log views · Session filtering |

This structure is *the* pillar-3 message ("The DevOps way for IBM PA") made concrete. Use it to drive section order on any Arc+ page.

## Common trap when writing Arc+ copy

Two traps.

**Trap 1 — forgetting Arc+ is a bundle.** New visitors confuse Arc+ as a third product. Always frame it as the connection between Arc and Pulse (at least once on any page).

**Trap 2 — leading with features instead of the lifecycle.** Arc+ has a lot of features. Listing them without the Plan → Build → Test → Migrate → Operate → Support arc makes it feel like a grab-bag. The arc is what makes Arc+ memorable.
